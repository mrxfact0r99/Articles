What is NCIIPC? 🤔

Before diving into the story, let’s talk about NCIIPC — The National Critical Information Infrastructure Protection Centre. It is the Indian government agency responsible for protecting the country’s Critical Information Infrastructure (CII), which includes servers, databases, banks, power plants, and telecom networks.

If you ever find a bug on a government website (ending with .gov.in), you can report it to the NCIIPC Responsible Vulnerability Disclosure Program (RVDP) via their official website: https://nciipc.gov.in/RVDP.html Based on the severity of the bug, they patch it and acknowledge your contribution. This is where I reported my findings, contributing to securing Digital India. 🇮🇳

Now, let me share my story of discovering two critical vulnerabilities on Indian government websites and how the government acknowledged my efforts. 🚀

First Bug 🐞

Recently, I found a high and medium-severity vulnerability on one of the Indian government websites, let’s call it “target.gov.in” for security reasons. The website was using WordPress, and as many security professionals know, WordPress sites can be vulnerable if not configured properly. I began testing for WordPress-related bugs, and soon, I found a critical one.

Vulnerability #1: XML-RPC Attack 🚧

The website had XML-RPC enabled, which is a feature in WordPress used for functions like pingbacks and trackbacks. However, when misused, it can lead to large-scale DDoS attacks and brute force attacks on login credentials. Here’s how I discovered it:

POST /xmlrpc.php HTTP/2
Host: target.gov.in
<methodCall>
<methodName>system.listMethods</methodName>
<params></params>
</methodCall>

The response I received confirmed that XML-RPC was enabled. This made the site vulnerable to DDoS attacks, as XML-RPC can be automated to send requests from multiple hosts, overwhelming the server. It could also be used to brute force admin credentials and gain unauthorized access to the website. 😱

Example of Brute Force Attack Request:

<methodCall>
<methodName>wp.getUsersBlogs</methodName>
<params>
<param><value>{your_username}</value></param>
<param><value>{your_password}</value></param>
</params>
</methodCall>

How to Fix It 🛠️

I suggested that the XML-RPC file should either be disabled or removed if not in use. At the very least, it should be blocked from external access to prevent such attacks.

After reporting this, the issue was acknowledged, and I was glad to contribute to securing the country’s digital infrastructure! 🙌

Second Bug 🐞

The second vulnerability I found was even more critical. This time, it was on an education-related government website that was leaking Personally Identifiable Information (PII) and sensitive documents, including exam papers.

Vulnerability 2: Directory Listing Exposure 🔍

While using Google Dorking techniques

site:*.gov.in indexof /

I discovered that one of the directories on the website was exposed to the public. Anyone could access the /wp-content/uploads/ folder and view or download the files stored there. This directory contained exam papers and documents with sensitive personal data.

Steps to Reproduce:

  • Open a web browser.
  • Navigate to the URL: https://target.gov.in/wp-content/uploads/.
  • You will see the entire list of files and folders within the uploads directory, including sensitive documents.

Impact 💥

The directory listing vulnerability exposed critical information that unauthorized users could access. This type of vulnerability can lead to:

  • Exposure of sensitive data (personal data, exam papers, etc.)
  • Potential data breaches
  • Information leakage for future attacks
  • Compliance violations (such as violating data protection regulations like GDPR)

How to Fix It 🛠️

To fix this, I recommended disabling directory listing by adding the following line to the website’s .htaccess file:

Options -Indexes

Additionally, I advised the site administrators to restrict access to sensitive directories and implement proper authentication to prevent unauthorized access.
